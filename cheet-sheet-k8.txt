KUBERNETES SERVICE vs DEPLOYMENT (HOW THEY CONNECT)

1. IMPORTANT RULE
Service does NOT read Deployment YAML files.
Service does NOT know file names.
Service only works with Pods using labels.

--------------------------------------------------

2. WHAT HAPPENS INTERNALLY

You apply Deployment YAML:
  kubectl apply -f deployment.yaml

Kubernetes creates:
  → Pods
  → Each Pod has labels

Example:
  Pod 1 → app=backend
  Pod 2 → app=frontend
  Pod 3 → app=payment

--------------------------------------------------

3. HOW SERVICE WORKS

Service has a selector:

  selector:
    app: backend

Kubernetes checks:

  "Find all Pods where app=backend"

--------------------------------------------------

4. RESULT

Service connects ONLY to Pods with:
  app=backend

It ignores:
  app=frontend
  app=payment

--------------------------------------------------

5. MULTIPLE DEPLOYMENTS CASE

Deployment A → app=backend
Deployment B → app=frontend
Deployment C → app=payment

Service selector:
  app=backend

Result:
  Service → connects ONLY to backend Pods

--------------------------------------------------

6. IMPORTANT WARNING

If multiple deployments use same label:

  Deployment A → app=backend
  Deployment B → app=backend

Then:
  Service will connect to BOTH ⚠️

--------------------------------------------------

7. ROLE OF "type" IN SERVICE

type: ClusterIP     → internal access
type: NodePort      → external via node IP
type: LoadBalancer  → external IP (cloud)

NOTE:
"type" does NOT affect pod selection

--------------------------------------------------

8. FINAL ONE-LINE MEMORY

Service → selects Pods using labels
NOT → Deployment
NOT → YAML file

--------------------------------------------------

9. VISUAL FLOW

Deployment → creates Pods → adds labels
Service → uses selector → finds matching Pods

--------------------------------------------------

# 🔥 1. Port vs TargetPort vs NodePort (VERY COMMON CONFUSION)

```txt
Service
  port: 80          → where Service receives traffic
  targetPort: 80    → where Pod/container listens
  nodePort: 30007   → external access port (NodePort only)
```

👉 Flow:

```txt
User → NodePort → Service(port) → Pod(targetPort)
```

---

# 🔥 2. Pod is the REAL thing (not Deployment)

👉 Important mindset:

```txt
Deployment → manages Pods
Service → connects to Pods
```

👉 Pods are the **actual running containers**

---

# 🔥 3. Labels are EVERYTHING

You already learned this, but go deeper:

👉 You can use multiple labels:

```yaml
labels:
  app: backend
  env: prod
```

Service can match:

```yaml
selector:
  app: backend
  env: prod
```

👉 This gives **fine control**

---

# 🔥 4. ReplicaSet (hidden but important)

```txt
Deployment → ReplicaSet → Pods
```

You don’t usually write ReplicaSet manually, but:

👉 It ensures:

```txt
"Always keep X number of Pods running"
```

---

# 🔥 5. What happens when a Pod dies?

👉 Kubernetes automatically:

```txt
Pod crashes → ReplicaSet creates new Pod → Service continues working
```

👉 This is **self-healing**

---

# 🔥 6. Rolling Updates (very important in real world)

When you change image:

```yaml
image: nginx:latest → nginx:v2
```

👉 Kubernetes does:

```txt
Old Pods ↓
New Pods ↑
(No downtime)
```
